# Daily Retro — 2026-03-16 (Monday)

## What Happened
No trades executed today. The gap check filter rejected all signals at open, resulting in a flat day with no P&L impact. Starting capital: $1,000.

This is the first trading day following the 2026-03-11 session where 3 trades were placed — all losses (NVDA put, BAC call, AAL call). The gap check was likely added in response to that session.

## What Worked
- Gap check filter is doing its job. All signals rejected = no new losses. Given the 0-for-6 historical win rate, sitting out is better than trading bad setups.
- Stop loss triggered correctly on 2026-03-11. AAL was stopped at -57.3% (target was -50%) — prevented further bleed.
- No overrides. The bot respected its own rules and did not force trades.

## What Did Not Work
- Win rate is 0/6 across all recorded trades. Every position logged in performance.json ended in a loss. Average loss: ~-34.9% per trade.
- Entry prices all recording as 0. This is a data integrity bug — the entry field is 0 on every trade. Makes true P&L calculation impossible (exit price is logged, but entry is missing).
- signal_type is "unknown" on every trade. Signal classification is not working — all sub-scores (options_score, news_score, politician_score) are 0. Signals are scoring 8-10 on signal_score but the sub-components are all zero. Something is broken in the scoring pipeline.
- Calls on bullish signals (AAPL Mar 3-4) failed. Two consecutive AAPL calls, both losers — suggests signal direction or timing was wrong.
- Put on AMZN (Mar 5) also failed. Directional accuracy appears poor across the board.

## Key Learnings
1. Data bugs are masking the real problem. Fix the entry price recording first — without it, cannot calculate actual P&L%, cannot evaluate signal quality, cannot improve. Priority 1.
2. signal_type = "unknown" means the scoring system is not feeding into trade selection properly. Sub-scores are all zero — either data sources are not running or scores are not being written to the trade record at execution time.
3. Gap check is a good defense but not a strategy. It prevented trades today (net positive), but root issue is signal quality. If signals were good, gap check would approve them.
4. 0-for-6 on paper trades is a signal, not noise. The strategy as currently implemented does not have edge. Need to fix the scoring pipeline or paper-test in simulation before live execution.
5. EOD closes are generating the losses. 5 of 6 trades closed at EOD — the bot either only runs at open/close or does not have working intraday exit logic.

## Adjustments for Tomorrow
- Fix entry price recording bug — entries are logging as 0. Find where the trade is written and capture actual fill price.
- Debug signal_type classification — trace why all trades show "unknown" and why sub-scores are all 0.
- Check if bot is running during market hours — if all closes are EOD, either the bot only runs at open/close or intraday stop logic is not triggering.
- Add a simulation/backtest mode — before placing new paper trades, validate the signal pipeline works correctly in simulation.
- Gap check is working, keep it. But log WHY signals are rejected (which gap threshold, which ticker) so we can tune it.
